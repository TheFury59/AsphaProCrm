# Aspha Pro — Lessons

Format : `[YYYY-MM-DD] | ce qui a mal tourné | règle pour l'éviter`

- [2026-05-12] | shadcn 4.x exige Tailwind v4 (installation initiale v3 a planté) | toujours vérifier la matrice de compat shadcn × Tailwind avant l'install
- [2026-05-12] | CORS `allowed_origins: ['*']` incompatible avec `supports_credentials: true` (browser rejette les cookies) | utiliser `allowed_origins_patterns: ['/.*/']` en dev avec credentials
- [2026-05-12] | `Relation::enforceMorphMap()` bloque Spatie ActivityLog (User fait des morphs auto) | utiliser `Relation::morphMap()` (sans enforce)
- [2026-05-12] | `DateTime::createFromFormat` retourne `false` avec Carbon → Factur-X cassé | utiliser `new \DateTime((string) $value)` quand la source est Carbon
- [2026-05-12] | `exception_date` stockée en datetime UTC (22:00Z = 18/05 Paris) ne matchait pas la clé YYYY-MM-DD | `Carbon::parse(...)->toDateString()` pour les clés d'exception
- [2026-05-12] | shape Eloquent `clientCompanies` HasMany alors qu'on attend `company` HasOne | patcher le model après génération automatique DBML, et propager le rename dans tous les eager loads
- [2026-05-12] | PowerShell `ConvertTo-Json` + `Invoke-WebRequest` encoding chelou en UTF-16 BOM | passer du JSON brut en string littérale plutôt que via hashtable convertie
- [2026-05-12] | Notifications table avait `notification_type_id`/`target_type`/`target_id` (pas `type`/`data`) | aligner systématiquement le type TypeScript sur la migration réelle, pas sur l'assomption "Laravel notifications standard"
- [2026-05-12] | Migration `make:migration create_vehicles_tables` génère un boilerplate `Schema::create('vehicles_tables')` qu'il faut écraser entièrement | toujours relire la migration générée avant de la sur-écrire, et utiliser `Read` avant `Write` (l'outil l'impose)
- [2026-05-12] | `MessageThreadParticipant` pas généré par les scripts DBML-to-models (composite pk) | créer manuellement les models pivot quand on doit en lire/écrire les colonnes indépendamment (cas last_read_at)
- [2026-05-12] | `MapPage` utilisait `addr.line1` / `addr.lat` qui n'existent pas | la migration Address réelle a `address`, `latitude`, `longitude` — toujours vérifier la migration avant d'écrire un composant qui consomme la ressource
- [2026-05-13] | Drag-and-drop FullCalendar décalait les RDV de 2h (15h→13h) à cause du round-trip TZ | NE PAS utiliser `Date.toISOString()` pour POSTer une datetime (toujours UTC) quand le backend est en `APP_TIMEZONE=Europe/Paris` + SQLite (qui perd la TZ). Toujours envoyer en local naïf `"YYYY-MM-DDTHH:MM:SS"` sans Z ni offset → Carbon parse en TZ applicative, round-trip stable.
- [2026-05-13] | Recalculer `occurrence_date` depuis la NOUVELLE date après drag d'une occurrence virtuelle | sinon l'exception est liée à l'ancienne date et n'efface pas la bonne occurrence. Utiliser `arg.event.start.toLocaleDateString("sv-SE")` (format ISO YYYY-MM-DD local).
- [2026-05-13] | `replace_all: true` sur `where('user_id', ...)` a aussi remplacé les occurrences sur `Employee` (qui doit garder `user_id`) | méfiance avec replace_all sur identifiants partiels — préférer des edits ciblés avec contexte unique quand 2 entités proches partagent le même champ
- [2026-05-13] | FullCalendar ne fournit pas de hook contextMenu | wrapper le calendar dans un `<div onContextMenu>` et lire `data-time`/`data-date` du slot ciblé pour récupérer la date du clic
- [2026-05-13] | Settings paramétrables : préférer table `app_settings` JSON-value typée à des constantes hardcodées | permet à l'admin de modifier seuils (45 min, 5j absence, etc.) sans déploiement. Cache 60s pour ne pas hit BDD à chaque request
- [2026-05-13] | Validation Laravel rejetait `employee_id: null` avec `validation.exists` | la règle `exists:` valide null aussi tant qu'on n'a pas ajouté `nullable` explicite. Toujours combiner `['sometimes', 'nullable', 'exists:...']` sur les FK optionnelles
- [2026-05-13] | `owner_user_id` sur table employees était NOT NULL sans default → violation contrainte lors d'une création | défaut auto = utilisateur courant côté controller pour les champs FK "gestionnaire" obligatoires
- [2026-05-13] | Géocodage BAN échouait silencieusement sur Windows (cURL error 60 SSL) | PHP Windows n'a pas bundle CA → `Http::withOptions(['verify' => false])` uniquement quand `app()->environment('local')`. Garder strict en prod Linux
- [2026-05-13] | `Cache::remember()` cachait aussi les échecs `null` pour 30j → adresse corrigée restait bloquée | gérer le cache manuellement (Cache::get/put) et NE PAS cacher les échecs, seulement les succès
- [2026-05-13] | `L.circle(center).getBounds()` crash avec `layerPointToLatLng undefined` quand le circle n'est pas attaché à une map active | calculer les bounds manuellement via formule géo (1° lat ≈ 111 km, 1° lng ≈ 111 km × cos(lat))
- [2026-05-13] | Cache invalidation React Query : mutation Intervention n'invalidait que `["interventions"]`, pas `["planning"]` → panneaux trajets/contrats stale | helper `invalidatePlanningRelated()` qui invalide en batch (interventions + planning + matching) sur create/update/delete/exception
- [2026-05-13] | DialogContent shadcn applique `sm:max-w-sm` par défaut → override `max-w-[1280px]` (sans préfixe) ignoré au-dessus de sm | utiliser `sm:!max-w-[1280px]` avec `!` (important Tailwind) pour forcer l'override
- [2026-05-13] | Rules of Hooks violées : `useMemo` après early returns conditionnels → "Rendered more hooks than during the previous render" | TOUS les hooks doivent être appelés AVANT tout early return, sans exception. Avec fallback (null/0) si les data ne sont pas chargées
- [2026-05-13] | Filter `where('status', 'active')` sur Employee alors que la colonne n'existe pas en BDD | toujours vérifier la migration avant d'écrire un filter. Sur SQLite l'erreur "no such column" arrive direct en 500 alors qu'en MySQL elle peut être silencieuse selon le mode strict
- [2026-05-13] | `<Circle>` Leaflet crash en mutation in-place du `radius` prop sur certains états transitoires | `key={radiusKm}` force un remount complet propre à chaque changement
- [2026-05-13] | ClientController/EmployeeController index ne faisaient pas `with('addresses')` → MapPage affichait "aucune adresse géocodée" alors que les coords étaient en BDD | toujours eager-load les relations utilisées en consommation directe sur les listes
- [2026-05-15] | Path frontend `client?.client_companies?.[0]?.company_name` sur 3 pages alors que le backend retourne `client.company` (singular HasOne) | quand on patche un model auto-généré DBML pour transformer HasMany en HasOne, propager la rename dans TOUS les consommateurs frontend (grep nécessaire car TS n'aide pas — `any` masque)
- [2026-05-15] | Devis créés sans items (pas de table quote_items mirror invoice_items) → devis "vides" | quand on copy-paste un pattern (Invoice items inline), créer aussi la migration table miroir + relation, pas juste le dialog UI
- [2026-05-15] | Cascade d'invalidation React Query : mutations addresses/contacts d'un client n'invalidaient pas le calendar (qui eager-load ces données) | helper `invalidateClientCascade` qui invalide `['client', id, sub]` + `['clients']` + `['interventions']` + `['planning']` en batch sur chaque mutation
- [2026-05-15] | `TypeError: rows.filter is not a function` sur QuotesListPage parce que le hook retournait la pagination Laravel double-wrappée (`['data' => $paginate]` + `paginate.data = [...]`) | toujours **normaliser** côté hook avec un helper `unwrapPaginated<T>()` qui détecte le double-wrap et retourne `{ data: T[], meta: {...} }` cohérent — sinon les composants finissent par accéder à un objet au lieu d'un array et plantent au runtime
- [2026-05-15] | `ClientMissionsTab` listait `client_prestations` à plat sans grouper par mission, alors que la BDD a une hiérarchie 1:N `missions → client_prestations` qui matérialise le contrat client | quand le schéma a une relation parent-enfant **structurante métier** (pas juste un FK de référence), l'UI doit la refléter dès le départ : cartes expandables parent + enfants imbriqués, dialogs séparés "créer parent" puis "ajouter enfant"
- [2026-05-15] | Avais codé la création de mission en simple Dialog alors que Xelya/Ximi a une PAGE dédiée avec batch create mission+prestations | quand l'utilisateur réfère un outil concurrent ("comme sur Xelya"), TOUJOURS relire le DBML/spec pour aligner exactement les champs et valeurs d'énum (ex: `payment_methods // 'check,sepa,card,transfer,cash'` — ne pas inventer ses propres labels). Dialog = ajout enfant à parent existant. Page = création parent + N enfants en transaction.
- [2026-05-18] | Saisir un ID humainement dans un champ texte est insupportable pour l'utilisateur ("super chiant") — pattern present dans 3 endroits (messagerie, tickets, missions) | toute selection d'entite (client, intervenant, mission, prestation) doit passer par un picker visuel avec photo/logo + recherche serveur ; les ID restent uniquement dans le payload api, jamais dans l'UI
- [2026-05-18] | Notifications affichees toutes identiques rendent impossible la priorisation visuelle | mapper notification_type.code -> icone + couleur + module dans `lib/notification-styles.ts`, border-left coloree + badge module + lien "Ouvrir" contextuel selon target_type ; eager-load la relation notificationType cote backend
- [2026-05-18] | URL d'image avec mise a jour : sans cache-buster le browser garde l'ancien fichier en cache local apres remplacement | accessor Laravel ajoute toujours `?v={updated_at.timestamp}` a l'URL du fichier ; le navigateur recharge a chaque update car la query string change
- [2026-05-18] | Double création de modules pour la même donnée (ici tickets sur fiche client ET page admin globale, 2 UI séparées) avec mismatch silencieux sur les types/champs → contenu perdu sans erreur visible | quand on découvre une feature dupliquée pour la même table, **vérifier mismatch validation backend ↔ payload frontend en priorité absolue** (les enums Laravel `in:...` rejettent les valeurs invalides mais sans message visible côté UX si la mutation a un onError générique). Unifier ensuite le flow : même endpoint OU même hook + cascade invalidation sur toutes les vues qui lisent la table
